# Проверка подписи в UGOS Pro: как устроена и можно ли подписывать самим

Разбор сделан по прошивке `firmware_image-1.16.0.89` (UGOS Pro, Debian 12, amd64).

## Кто проверяет пакеты

- **`/usr/sbin/uginstall`** — установщик пакетов `.upk` (Go-бинарник). Именно он
  разбирает контейнер `UGREEN-PKG-V2-FORMAT`, проверяет подписи и разворачивает
  дерево приложения. Строки в бинарнике: `systemVerify`, `GetVerifyKey`,
  `verifyData`, `un_support_sign_ver`, `parse_package_error`.
- Криптография: **RSA + SHA-256** (в бинарнике — `crypto/rsa`, `crypto/sha256`,
  признаки RSA-PSS). То же подтверждает `upk_pack.py`: подписи `filesig/usersig/midsig`.

## Корень доверия (что «вшито» в прошивку)

Доверенные **публичные** ключи лежат в read-only разделе прошивки:

```
/rom/usr/ugreen/etc/rsa/ugreenRoot.pub         # корневой ключ
/rom/usr/ugreen/etc/rsa/ugreenRootImport.pub   # корневой ключ для импорта
/rom/usr/ugreen/etc/rsa/ugreen.pub             # ключ уровня приложения
```

`uginstall` читает `ugreenRoot.pub` как якорь доверия. Раздел `/rom` — это
`fw.squashfs` (сжатый xz squashfs, только для чтения), он часть подписанной прошивки.

## Цепочка подписи

Сопоставление ключей по 19 пакетам показывает трёхуровневую цепочку:

```
ugreenRoot (вшит в /rom, приватный ключ у UGREEN)
      └── подписывает промежуточный ключ канала  (midpub — ОДИНАКОВ во всех пакетах)
                └── подписывает ключ сборки       (userpub — СВОЙ у каждого пакета)
                          └── подписывает payload  (filesig/usersig)
```

- `midpub` во всех пакетах одинаков → это промежуточный ключ издателя.
- `userpub` уникален для каждого пакета → ключ конкретной сборки.
- В пакете хранятся только публичные части (`midpub`, `userpub`) и подписи.

## Главный вывод про «подписать самим»

**В прошивке есть только ПУБЛИЧНЫЕ ключи. Приватных ключей UGREEN там нет** — и не
должно быть (иначе это компрометация самого вендора). Поэтому:

| Вопрос | Ответ |
|---|---|
| Вытащить приватный ключ подписи с NAS? | **Нет** — его там нет, только `.pub`. |
| Подписать модифицированный пакет для «стокового» NAS? | **Нет** — нужен приватный ключ `ugreenRoot`/`mid`. Подделать = взломать RSA. |
| Подписать своим ключом на своём устройстве? | Технически да, но только заменив якорь доверия (см. ниже). Хрупко, не рекомендуется. |

### Почему «своя подпись» — плохой путь

Чтобы NAS принял пакет, подписанный вашим ключом, нужно заменить
`ugreenRoot.pub` (и связанные) на ваш публичный ключ. Но:

- `/rom` — read-only squashfs из **подписанной** прошивки; прошивку проверяет
  отдельный апдейтер `ugupdate`, так что просто пересобрать прошивку не выйдет;
- подмена ключа во время работы (bind-mount поверх `/rom`) не переживёт обновление
  системы и трогает системные компоненты;
- это ломает штатную проверку подлинности всех приложений на устройстве.

## Когда проверка вообще выполняется

- **Подпись проверяется только при УСТАНОВКЕ** пакета (`uginstall`).
- После установки в файловой системе остаётся только дерево приложения
  (`contents/`), самого `.upk` и подписей там уже нет.
- **Во время работы** UGOS использует режим `reconcile`: проверяет пропажу файлов
  (`reason=files-missing`) и манифест целостности **`.check-app`** (JSON
  `путь → md5`). Подпись при этом заново НЕ проверяется.

## Практический вывод для локализации

Раз подпись проверяется только при установке, а приложение потом работает из папки
на диске — **свою локализацию добавляют поверх установленного приложения, без
переподписи**. Достаточно отредактировать файлы локали и обновить `.check-app`
(чтобы прошёл рантайм-`reconcile`). Готовые скрипты и инструкция —
в [LOCALIZATION.md](LOCALIZATION.md).
